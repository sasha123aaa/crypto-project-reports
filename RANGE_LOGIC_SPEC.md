# RANGE_LOGIC_SPEC.md

## Назначение

Этот документ фиксирует, как должна работать логика диапазонов для блока **«Быстрый теханализ»** в проекте `crypto-project-reports`.

Цель:
- не использовать упрощенную логику по одной свече
- не подменять range logic обычным трендом
- для каждого таймфрейма определять состояние через **последний активный диапазон**
- использовать одну и ту же логику для любой монеты из отчета, а не только для BTC

---

## Что является источником истины

Базовая логика уже есть в загруженном HTML-файле `market_dashboard (6).html`:
- используется функция `detectRangesWithPreview(...)`
- итоговое состояние берется из `previewRange[4]`
- если `previewRange[4] === true` → bullish
- если `previewRange[4] === false` → bearish
- если `previewRange` нет или она невалидна → neutral

Это ядро логики нужно считать основным референсом. fileciteturn5file0

---

## Что сейчас в файле правильно

1. Логика не сводится к `close > open`.
2. Есть поиск структуры диапазона через:
   - A
   - B
   - C
   - correction
   - expand mode
   - reversal
3. В конце рассчитывается `previewRange`.
4. Финальный bias строится по последнему активному диапазону. fileciteturn5file0

## Что сейчас в файле неправильно или опасно

### 1. Жестко зашит BTC
В dashboard включен режим:
- `USE_STATIC_SYMBOLS = true`
- `STATIC_SYMBOLS = ['BTCUSDT']`

Для проекта отчетов это неверно.
Символ должен браться из текущего проекта:
- ETH → `ETHUSDT`
- SOL → `SOLUSDT`
- LINK → `LINKUSDT`
- и так далее

Range logic должна работать не на BTC по умолчанию, а на монете текущего разбора. fileciteturn5file2

### 2. Потеряна глубина интерпретации
Сейчас на выходе логика дает только:
- bullish
- bearish
- neutral

То есть ядро диапазонов сохранено, но верхний слой анализа сильно упрощен.
Это допустимо как первый уровень, но нельзя называть это полной аналитикой диапазонов.

### 3. В ETH quick TA уже есть дополнительное упрощение
В текущем отчете итог по LTF / MTF / HTF строится через суммирование простых состояний по таймфреймам, а затем через score пишется короткий текст.
Это уже слой интерпретации, а не сама range logic. fileciteturn5file11

## Правильная модель работы

### Шаг 1. Берем символ текущего проекта
Нельзя использовать BTC как жесткий дефолт.
Нужно брать:
- `project.bybitSymbol`
или аналогичное поле конфигурации.

### Шаг 2. По каждому таймфрейму грузим историю свечей
Таймфреймы:
- 1m
- 3m
- 5m
- 15m
- 1h
- 4h
- 1d
- 1w

Интервалы Bybit:
- 1m → `1`
- 3m → `3`
- 5m → `5`
- 15m → `15`
- 1h → `60`
- 4h → `240`
- 1d → `D`
- 1w → `W` fileciteturn5file2

### Шаг 3. Для каждого ТФ применяем `detectRangesWithPreview`
На вход функции подается массив свечей в **хронологическом порядке**.

Каждая свеча должна иметь:
- `open`
- `high`
- `low`
- `close`
- `time`

### Шаг 4. Финальный state ТФ
После вызова `detectRangesWithPreview(candles, 0.3, 20, 7)`:

- если `previewRange` не существует → `neutral`
- если `previewRange[4] === true` → `bullish`
- если `previewRange[4] === false` → `bearish`

Именно это состояние показывается в quick TA. fileciteturn5file5

## Параметры логики

Текущие рабочие параметры из файла:

- `correctionPct = 0.3`
- `maxRects = 20`
- `minBars = 7` fileciteturn5file5

### Значение параметров

#### correctionPct = 0.3
Коррекция считается значимой, когда цена откатила примерно 30% от диапазона.

#### maxRects = 20
Храним ограниченное число диапазонов.
Это влияет на память и историю, но не должно ломать preview logic.

#### minBars = 7
Минимальная длина диапазона.
Если диапазон слишком короткий, он не должен считаться полноценным.

## Что Codex нельзя делать

1. Нельзя заменять эту логику на:
   - EMA crossover
   - close/open candle direction
   - slope по нескольким свечам
   - RSI / MACD / другие индикаторы

2. Нельзя писать «bullish» только потому, что последняя свеча зеленая.

3. Нельзя подменять range logic простой логикой:
   - higher high / higher low без остального алгоритма
   - breakout = bullish без подтверждения через previewRange

4. Нельзя молча менять `correctionPct` и `minBars`, если это не отдельная задача.

## Требование к реализации в репозитории

В проекте должно быть так:

- одна общая функция range logic
- один общий helper для получения bias по таймфрейму
- quick TA использует этот helper
- символ берется из `project config`
- fallback в случае ошибки = `neutral`
- источник котировок = Bybit spot, пока не задан другой

## Минимальный псевдопоток

1. определить `symbol` из проекта  
2. загрузить свечи по каждому ТФ  
3. прогнать через `detectRangesWithPreview`  
4. получить `previewRange[4]`  
5. конвертировать в:
   - bullish
   - bearish
